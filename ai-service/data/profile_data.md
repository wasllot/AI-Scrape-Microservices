## Patrones de Diseño Implementados

### 1. Factory Pattern (`providers_factory.py`)
Implementación de un **Abstract Factory** para la creación de proveedores LLM. Permite registrar proveedores custom para testing y seleccionar entre Gemini, Groq o Static en runtime sin modificar el código consumidor.

### 2. Strategy Pattern + Chain of Responsibility (`router.py`)
El **LLM Router** implementa una cadena de responsabilidad con fallback automático:
- **Primario**: Gemini API → si falla →
- **Secundario**: Groq (Llama 3) → si falla →
- **Fallback**: Respuesta estática (nunca retorna error 500)

### 3. Circuit Breaker Pattern (Redis-backed)
Implementación productiva del Circuit Breaker con tres estados (**CLOSED → OPEN → HALF_OPEN**) persistido en Redis. Previene llamadas continuas a proveedores que están fallando, con cooldown automático de 2 minutos.

### 4. Singleton Pattern (`MonitoringCollector`)
El `MetricsCollector` usa Singleton thread-safe con doble verificación de bloqueo para garantizar una única instancia de recolección de métricas en toda la aplicación.

### 5. Dependency Injection (FastAPI + Laravel)
- **FastAPI**: Inyección via `Depends()` para servicios de embeddings, chat y autenticación.
- **Laravel**: Binding de interfaces (`Contracts/`) a implementaciones concretas (`Services/`) en el Service Container.

### 6. Repository Pattern (Database Layer)
Abstracción de la capa de datos con `DatabaseConnection` que encapsula conexiones, pool management y ejecución de queries parametrizadas contra PostgreSQL.

### 7. Decorator Pattern (`@track_request`)
Decoradores para métricas que envuelven endpoints con tracking automático de latencia, contadores de requests y contadores de errores, sin modificar la lógica de negocio.

### 8. Interface Segregation (Laravel Contracts)
Interfaces PHP segregadas: `AIServiceInterface`, `HttpClientInterface`, `ScraperServiceInterface` — cada una con responsabilidad mínima y clara.

## Metodologías y Prácticas de Ingeniería

### Principios SOLID Aplicados
- **SRP**: Cada microservicio = una responsabilidad. Cada módulo Python = una función clara.
- **OCP**: Nuevos proveedores de IA se agregan sin tocar el router existente (registrar y listo).
- **LSP**: Todos los LLM providers implementan la misma interfaz `LLMProvider`.
- **ISP**: Interfaces PHP segregadas (3 contratos separados, no un "mega-interface").
- **DIP**: Dependencias inyectadas, nunca instanciadas directamente en la lógica de negocio.

### Clean Code & Mejores Prácticas
- **Docstrings** exhaustivos en todas las funciones y clases con formato NumPy/Google.
- **Type Hints** nativos en Python y **Typed Properties** en PHP.
- **Configuration Management**: Variables de entorno centralizadas con validación en Pydantic.
- **Error Handling**: Excepciones personalizadas con contexto, nunca errores genéricos al usuario.
- **Input Sanitization**: Todo input de usuario es sanitizado antes de procesarse (XSS, SQL Injection).

### Testing Strategy
- **Unit Tests**: Cobertura con pytest (Python) y PHPUnit (PHP).
- **Integration Tests**: Verificación de flujos completos entre servicios.
- **Mocking**: pytest-mock y Mockery para aislar dependencias externas (APIs de IA, DB).
- **Coverage Reports**: Generación de reportes HTML con pytest-cov.
- **Async Testing**: pytest-asyncio para validar flujos asíncronos de FastAPI.

### DevOps & Infrastructure as Code
- **Docker-First**: Todos los servicios corren en contenedores desde el día 1.
- **Health Checks**: Cada servicio expone probes de salud para orquestación.
- **Service Discovery**: Traefik descubre servicios Docker automáticamente via labels.
- **Database Migrations**: Scripts SQL idempotentes (`IF NOT EXISTS`) ejecutados al startup.

### Sobre Arquitectura y Diseño

**¿Cuándo elegirías microservicios en lugar de un monolito?**
Solo cuando la complejidad del dominio, la necesidad de escalabilidad independiente o la estructura organizacional lo justifiquen. Siempre empiezo con un monolito modular y extraigo servicios cuando hay una razón clara.

**¿Cómo manejas la comunicación entre microservicios?**
En este proyecto uso comunicación síncrona via HTTP/REST con clientes tipados. Para sistemas de mayor escala, implementaría colas de mensajes (RabbitMQ/Kafka) para comunicación asíncrona y event-driven.

**¿Qué patrones de diseño usas más frecuentemente?**
Factory, Strategy, Repository, Dependency Injection y Circuit Breaker. Cada uno resuelve un problema específico: el Factory para creación flexible, Strategy para algoritmos intercambiables, y Circuit Breaker para resiliencia.

**¿Cómo garantizas la resiliencia de un sistema distribuido?**
Implemento Circuit Breakers con Redis, retries con backoff exponencial (tenacity), fallbacks estáticos, health checks y monitoring con correlation IDs para trazabilidad end-to-end.

### Sobre Tecnologías Específicas

**¿Por qué FastAPI y no Django REST Framework?**
FastAPI ofrece mejor rendimiento (async nativo), validación automática con Pydantic, documentación OpenAPI generada automáticamente y es ideal para microservicios ligeros. Django REST es excelente para aplicaciones monolíticas más complejas.

**¿Qué experiencia tienes con bases de datos vectoriales?**
Implementé el motor de búsqueda semántica de este proyecto usando PostgreSQL + pgvector con embeddings de 3072 dimensiones generados por Gemini. Incluye ingesta de documentos, chunking inteligente y búsqueda por similitud coseno.

**¿Cómo implementas autenticación en microservicios?**
JWT stateless con tokens de servicio de larga duración para comunicación inter-servicio y tokens de corta duración para usuarios. La verificación es descentralizada — cada servicio valida el token independientemente.

**¿Qué experiencia tienes con Docker en producción?**
Uso Docker Compose para orquestación multi-servicio con healthchecks, volúmenes persistentes, redes aisladas y Traefik como reverse proxy con SSL automático. Tengo experiencia gestionando servidores de producción.
